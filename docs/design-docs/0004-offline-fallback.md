# ADR 0004: オフライン・取得失敗時のフォールバック

- 状態: 採用
- 日付: 2026-06-14

## 決定

- **既存プールは常に保持する**。補充が失敗(オフライン・タイムアウト・空レスポンス)しても旧プールを壊さず、そこから表示を続ける。
- **初回起動でプールが空のとき用に、同梱シード写真(`seed_photos.dart` + バンドル画像数枚)を持つ**。これにより初回オフラインでも必ず 1 枚表示できる。
- 補充失敗はエラー画面にせず、静かに「補充なし」とする(必要なら控えめなトーストで「オフライン観測モード」を示す。プロトタイプ準拠)。

## 背景

待ち受けアプリは圏外・機内モード・API 不調でも表示できなければならない(`docs/product-spec.md` のオフライン要件)。問題になるのは次の 2 局面:

1. **初回起動でプールがまだ空**かつオフライン → 表示するものがない。
2. **2 回目以降で補充に失敗** → 新しい写真は得られないが、既存プールはある。

プロトタイプはオフライン時に手続き的(canvas 生成)なフォールバック画像を合成しているが、本実装では扱いを整理する。

| 局面 | 候補 | 評価 |
| --- | --- | --- |
| 初回・空・オフライン | 同梱シード画像 | 確実に表示できる。アプリ容量は数枚分増える |
| 初回・空・オフライン | 手続き的生成(プロトタイプ) | 容量ゼロだが「本物の宇宙写真」ではない。実装も重い |
| 補充失敗 | 旧プール維持 | 体験が途切れない。最も素直 |
| 補充失敗 | エラー表示 | 待ち受けとして不適切 |

## 理由

- 「真っ白を避ける」が最優先。同梱シードなら初回オフラインでも本物の写真で成立する。
- 旧プール維持は実装も理解も単純で、プール方式(ADR 0001)と一貫する。
- 手続き的生成はプロトタイプのデモ用途には有効だったが、本実装ではシード画像で代替し、コードの複雑さを持ち込まない(必要になれば再検討)。

## 結果

- `data/seed_photos.dart` に同梱写真のメタ + バンドル画像参照を定義する。プールが空のときだけ使い、補充が成功したら通常プールに置き換わる。
- 同梱画像のライセンス(NASA クレジット)を `assets/` 配下に明示する。
- 補充経路は失敗を握りつぶして旧プール(または シード)を残す。例外でアプリを落とさない。
- `PoolStore` から読んだプールが壊れていた場合も、例外にせず空扱い → シードで起動する(永続化スキーマのフォールバック方針、`docs/test-plan.md` の unit test 対象)。
