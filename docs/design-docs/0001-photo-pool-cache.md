# ADR 0001: 写真のローカルプール方式と補充戦略

- 状態: 採用
- 日付: 2026-06-14

## 決定

写真は**端末ローカルのプール(数十〜数百枚)からランダムに 1 枚選んで表示**する。

- **起動時はプールからの選択のみ。ネットを叩かない**(同期・オフライン可・高速)。
- **補充は前回更新から 24 時間以上経ったときだけ、起動後にバックグラウンド(フォアグラウンド非同期)で**行う。判定は `RefreshPolicy`(`Clock` 注入、pure Dart)、取得は `PhotoSource` 境界。
- プールのメタデータは `PoolStore`、画像バイナリは `ImageStore` に永続化する。
- 選択の乱数は `Random(seed)` を注入する。

## 背景

`docs/product-spec.md` の中心要件「毎回ちがう写真」をどう実現するか。候補:

| 候補 | 評価 |
| --- | --- |
| 起動のたびに API を叩く(プロトタイプ方式) | 実装は単純だが、圏外で真っ白・起動が遅い・レートリミットの懸念。待ち受けアプリとして致命的 |
| ローカルプール + 低頻度補充 | 表示は通信ゼロでオフライン可・高速。通信は 1 日 1 回程度に圧縮。実装はプール管理とキャッシュが要る |
| 全画像を初回に一括 DL して固定 | 「毎回新鮮」が薄れる。容量も読みづらい |

## 理由

- 待ち受けは**オフラインで確実に表示できること**が最優先(機内モード・圏外)。プール方式なら表示経路に通信がない。
- **起動が速い**(通信待ちなし)。
- API レートリミットと無縁(通信は補充時の 1 日 1 回程度)。
- 「プールからランダム選択」だけで体感は十分に「毎回ちがう」。補充でプールを入れ替えれば長期的にも新鮮さを保てる。

## 結果

- 表示経路(`ViewerController`)は `PhotoSource` を呼ばない。**通信は補充経路(`PoolController` / `PoolRefresher`)に限定**し、テストで「表示時に source が呼ばれない」ことを assert する(`docs/test-plan.md`)。
- `RefreshPolicy.shouldRefresh(now, lastRefreshedAt)`: `lastRefreshedAt == null` または `now - lastRefreshedAt >= 24h` で true。間隔(24h)は定数化し、後から調整可能にする。
- **補充に失敗しても旧プールを保持**する(オフライン・API 不調でアプリが壊れない。ADR 0004)。プール件数の下限を定め、それを割る入れ替えはしない。
- プールの保存形式・画像キャッシュの実体は `PoolStore` / `ImageStore` 境界の裏に隠す。容量上限・古い画像の追い出し(LRU 等)が必要になったら境界実装を差し替え、新しい ADR を書く。
- **プールはカテゴリ(観測テーマ)別に持つ**。`PoolStore` は `load(category)` / `save(category)` でカテゴリ単位に読み書きし、補充も 24h 判定をカテゴリ単位で行う。テーマ切替時は、そのカテゴリのローカルプールを即表示してから(通信なし)、必要なら補充する。
- 画像の追い出し(`ImageStore.retainOnly`)は**全カテゴリのプールの和集合**を keep 対象にしないと他カテゴリの画像を誤削除する。補充経路(単一カテゴリしか知らない)からは行わず、全カテゴリを束ねる `ImageCacheCleaner`(`domain/pool/image_cache_cleaner.dart`)が担う。**未ロードのカテゴリも `PoolStore` から読んで union に含める**。`PoolController` が補充成功後にベストエフォートで `prune()` を呼ぶ。容量上限(LRU 等)が必要になったら別 ADR。
- 補充の実行枠を OS のバックグラウンド実行(WorkManager / BGTaskScheduler)へ広げるかは未決(`docs/harness-engineering.md` 未決質問)。初期は「起動後フォアグラウンド非同期」で始める。
